> 测评前先清空之前的测试数据，确保干净环境

## 二、V1 核心路径（用例 1-6）

### 用例 1：纯想法 → 完整追问 → 笔记生成

**目的：** 验证 Router → Socratic 追问 → 收尾出笔记（含标签）

**发送：**
> 我觉得很多人学东西只停留在「收藏」，从来不回看

**检查项：**
- [ ] Bot 回复第 1 个追问，语气自然，像朋友聊天
- [ ] 不列选项（不出现「是 A、B 还是 C？」）
- [ ] 你回复后，Bot 继续追问（最多 3 轮）
- [ ] 收尾消息包含 ✅ + 笔记格式
- [ ] 笔记包含「标签」字段（3-5 个标签）← V2 新增
- [ ] 笔记日期、洞察、原始触发、追问摘要都有内容

**你的回复参考：**
- 第 1 轮：「就是我自己，微信收藏了一堆文章，从来没打开过」
- 第 2 轮：「比如上次看到一篇 AI PM 能力模型的文章，收藏了一直没看，面试被问到完全没准备」

**记录：** 追问了几轮？哪句最自然/最生硬？标签准不准？

---

### 用例 2：带 URL + 看法 → 跳过锚定

**目的：** 验证混合消息快捷判断（URL + 看法 → 跳过锚定直接深挖）

**发送：** 找一篇你看过的文章，附带看法。比如：
> https://mp.weixin.qq.com/s/xxxxx 这篇讲 AI 时代产品经理的转型，我觉得说到点子上了

**检查项：**
- [ ] Bot 没有问「你最想搞清楚哪一点」（跳过了锚定）
- [ ] Bot 直接围绕你的看法深挖
- [ ] 收尾笔记标签准确

---

### 用例 3：主动结束对话

**目的：** 验证 conversation_end 路由

**发送：**
> 最近觉得写作其实是一种思考工具

**Bot 追问后直接说：**
> 先到这里吧

**检查项：**
- [ ] Bot 识别结束语，不再追问
- [ ] 回复包含总结 + 笔记格式
- [ ] 笔记「个人应用场景」写的是「待补充」（因为没深入聊）

---

### 用例 4：提前收尾（给出具体场景）

**目的：** 验证具体应用场景触发提前收尾

**发送：**
> 我发现跟别人解释一个概念的时候，自己反而理解更深了

**第 1 轮就给具体场景：**
> 上周给同事讲广告投放的归因模型，讲着讲着自己突然想通了一个一直没搞清楚的逻辑

**检查项：**
- [ ] Bot 识别到具体场景，不再凑满 3 轮
- [ ] 笔记「个人应用场景」有内容（不是「待补充」）

---

### 用例 5：/stats 指令

**发送：**
> /stats

**检查项：**
- [ ] 返回统计数据（对话总数、完成率等）
- [ ] 数据和本地 `python3 sync-notes.py --stats` 大致一致

## 三、V2 收藏加工器（用例 7-10）

### 用例 7：收藏入库（只发链接）

**目的：** 验证 bookmark 路由——只发链接不带想法时，Bot 静默入库

**发送：**（随便找一篇文章链接，不要附带任何看法）
> https://mp.weixin.qq.com/s/xxxxx

**检查项：**
- [ ] Bot 回复「收到，已加入待消化。[一句话摘要]」
- [ ] Bot 没有追问
- [ ] 回复中包含对内容的简短描述（说明 Bot 读了这篇文章）
- [ ] Bot 没有暴露抓取过程（不说「正在获取链接」之类的话）

---

### 用例 7b：链接抓取失败 fallback（Plan 8 新增）

**目的：** 验证链接抓取失败时的优雅降级

**发送：**（找一个小红书或需要登录才能看的链接）
> https://www.xiaohongshu.com/explore/xxx

**检查项：**
- [ ] Bot 回复「收到，已加入待消化。链接我暂时打不开，下次消化时你可以补充一下内容。」
- [ ] 或者 Bot 回复 fallback 提示让用户贴内容
- [ ] Bot 不报错、不卡死
- [ ] 这条收藏仍然入库（运行 sync-notes.py 后 pending_digests 有记录）

---

### 用例 8：收藏 vs 消化的区分

**目的：** 验证同样是链接，有看法 → 追问，没看法 → 收藏

**先发（无看法）：**
> https://另一篇文章链接

**预期：** 「收到，已加入待消化」

**再发（有看法）：**
> https://又一篇文章链接 这个观点我不太同意

**预期：** Bot 开始追问（走 content_digest 路径）

**检查项：**
- [ ] 无看法 → bookmark 路由（不追问）
- [ ] 有看法 → content_digest 路由（追问）
- [ ] 两种情况的区分准确

---

### 用例 9：/pending 和 /digest 指令

**前提：** 用例 7、8 已经收藏了至少 1 条内容

**发送：**
> /pending

**检查项：**
- [ ] 返回待消化收藏列表
- [ ] 按主题分组
- [ ] 包含每条收藏的摘要
- [ ] 结尾问「想从哪个方向开始聊？」

**然后发：**
> /digest

**检查项：**
- [ ] 效果和 /pending 相同

---

### 用例 10：被动关联（对话中提到相关收藏）

**前提：** 用例 7-8 已收藏了内容，并且你现在聊一个相关话题

**发送一个跟之前收藏内容相关的想法，完成追问后：**

**检查项：**
- [ ] 笔记末尾是否提到「你之前收藏了一篇关于 XX 的内容」
- [ ] 提到的内容确实和当前话题相关（不是乱关联）
- [ ] 如果没有相关收藏，就不提（不强行凑）

> 注：这个用例依赖标签重合，如果收藏太少可能不触发，属于正常。

## 四、同步验证（用例 11-12）

**在终端运行：**


**检查项：**
- [ ] 输出显示新增笔记数量

- [ ] 输出显示收藏识别数量
- [ ] `~/Documents/ObsidianVault/Crucible/` 目录下有新 .md 文件

### 用例 12：Obsidian 中验证

**打开 Obsidian：**
- [ ] Crucible 文件夹下能看到新笔记
- [ ] 笔记 YAML frontmatter 中 tags 包含语义标签（不只是 crucible）
- [ ] 如果有关联笔记，底部有 `[[双链]]` 格式的关联
- [ ] 点击双链能跳转到对应笔记
