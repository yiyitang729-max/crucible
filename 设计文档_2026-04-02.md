## 一、产品定位与目标用户

### 1.1 产品定位

Crucible 是一个 **AI 思考伙伴**，专为在移动端大量消费内容却难以沉淀输出的知识工作者设计。

它解决的不是"存不进去"的问题——市面上的收藏工具已经足够多——而是"存进去之后什么都没发生"的问题。Crucible 通过对话式 AI 追问，帮助用户把每一个碎片想法从"模糊感受"变成"清晰认知"，并自动沉淀为可检索、可关联、可生长的个人知识库。

**一句话**：让每次碎片输入不再消失，通过 AI 追问变深，自动沉淀为 Obsidian 里的结构化知识。

### 1.2 目标用户

**核心用户画像**

- 每天在手机上大量消费内容（Twitter/X、公众号、小红书、YouTube）
- 有收藏习惯，但收藏夹是"内容坟场"——越积越焦虑
- 有想法，但完美主义导致不敢整理输出（没想完整就不动笔）
- 认可 Obsidian 的知识关联能力，但实际使用率低
- 已经在把文章截图发给 AI 问理解，但理解停在聊天记录里消失

**职业特征**：知识工作者（产品经理、分析师、研究者、内容创作者）

**关键行为特征**：已有把内容分享给 AI 的习惯，缺的是让这个行为产生积累的机制。

### 1.3 核心价值主张

| 用户痛点 | Crucible 的解法 |
|---------|---------------|
| 收藏了从不回看 | 输入即触发对话，强制当下处理 |
| 理解停在 AI 聊天里消失 | 对话产出自动沉淀为 Obsidian 原子笔记 |
| 碎片知识无法串联 | 向量检索自动关联历史笔记，建立双链 |
| 完美主义阻碍输出 | 低门槛入口，任何碎片都被接住；输出从一句话开始 |

### 1.4 与竞品的核心差异

市场上的工具普遍解决 **Capture + Organize**（存进去、找得到）。

Crucible 的位置在 **Distill**——帮用户把模糊感受变成清晰认知。这一层目前是市场真空。

```
竞品覆盖：  [Capture] → [Organize] → ___Distill___ → [Express]
Crucible：  [Capture] → [Organize] → [Distill ✓] → [Express（V3）]
```

### 2.2 各版本定义

**V1 — MVP：跑通核心追问闭环**

> 本设计文档描述 V1 的完整设计。

**迭代点**：从零到一，让"一次追问 → 一条有价值的笔记"这个最小闭环跑通。V1 之前用户把内容发给 AI，理解消失在聊天记录里；V1 之后，每次对话产出的洞察会自动沉淀为 Obsidian 笔记，不再消失。

核心功能：
- 飞书 Bot 接收任意输入（文本 / URL / 截图）
- Socratic Agent 追问对话（2-4 轮），引导用户找到个人应用场景
- 自动生成原子笔记写入 Obsidian，格式统一、包含洞察摘要

验收标准：用户发一条内容，走完追问，Obsidian 出现一条包含个人洞察的笔记。

---

**V2 — 知识网络成型（已完成）**

**迭代点**：从"一条条孤立笔记"到"相互关联的知识网络"。V1 的笔记是孤岛；V2 通过标签系统让笔记自动关联，同时增加收藏加工机制，解决"收藏了不看"的焦虑。

> **架构决策变更（v0.3）**：原计划用 LanceDB 向量检索做笔记关联，实际开发中改为**标签系统 + TAG_INDEX**方案。原因：V2 阶段笔记数量少（<100 条），向量检索的优势在数据量小时不明显，标签系统更轻量、可解释性更强、用户能直观理解关联逻辑。向量检索作为 V∞ 储备方案保留。

实际完成功能（Plan 5-7）：
- 标签系统：AI 自动生成 3-5 个语义标签，构建 TAG_INDEX.json
- 笔记关联：标签重合 ≥2 的笔记自动建立 Obsidian 双链 `[[]]`
- 收藏入库：只发链接不带想法 → 静默收藏到 pending_digests 表
- 收藏加工：/pending、/digest 指令唤起待消化列表，按主题聚类推荐
- 被动关联：对话收尾时自动检查是否有相关待消化收藏
- 心跳提醒：定期检查未消化收藏，主动推送提醒
- 路由增强：从 3 种意图扩展到 6 种（新增 bookmark、command、conversation_end）

---

**V3 — 闭合输出闭环**

**迭代点**：从"知识存进去"到"知识能产出来"，完整闭合 CODE 的 Express 阶段。V2 积累了足够的原子笔记；V3 让 AI 把相关笔记聚合成草稿，降低输出的启动摩擦，真正解决"只输入不输出"的问题。

新增功能：
- Express 草稿生成：AI 聚合相关笔记，生成可编辑的输出草稿
- 输出格式渐进：Obsidian 笔记 → 推文 → 公众号/小红书文章
- 多平台采集入口：浏览器插件 / iOS Share Sheet，减少切换摩擦

---

**解耦 — 核心逻辑独立于 Harness（进行中）**

**迭代点**：将 Agent 核心逻辑（追问策略、路由规则、身份定义）从 OpenClaw 平台解耦，使其成为独立的、可移植的模块。

> **什么是 Harness？** Harness 是 Agent 的「管理壳」，负责：Agent 生命周期管理、上下文管理、安全护栏、评估/可观测。OpenClaw 在 Crucible 中充当 Harness 角色——管理消息路由、会话状态、工具调用。解耦的目标不是替换 OpenClaw，而是让核心 Agent 逻辑不依赖特定 Harness，未来可接入 Web 前端、企业微信、Claude Agent SDK 等不同运行时。

目标架构：
```
┌─────────────────────────────────────┐
│          agent_core/（核心层）        │
│  SOUL.md     — 追问流程 + 路由规则    │

│  USER.md     — 用户画像              │
│  HEARTBEAT.md — 定时任务规则          │
└──────────┬─────────────┬────────────┘
    ┌──────▼──────┐  ┌───▼──────────┐
    │ adapters/   │  │ adapters/    │
    │ openclaw/   │  │ web/ (未来)  │
    │ (当前使用)   │  │              │
    └──────┬──────┘  └───┬──────────┘
    ┌──────▼──────┐  ┌───▼──────────┐
    │  飞书 Bot    │  │  浏览器/API   │
    └─────────────┘  └──────────────┘
```

**V∞ — 长期愿景：Distill Engine + To B 扩展**

### Distill Engine（蒸馏引擎）——产品核心竞争力

**核心理念**：Distill 的本质不是给用户新信息（那是 ChatGPT 做的事），而是**从用户已有的认知里，提取他自己没意识到的模式、没经过验证的假设、没闭环的想法**。

市面上的 AI 工具都在顺着用户说。Crucible 的 Distill Engine 做的是：帮用户看清自己已经知道但没想明白的东西。

**从浅层到深层的演进**：

```
V1-V2（浅层 Distill）              V∞（Distill Engine）
单次对话内追问 2-3 轮               跨对话、跨时间的深度认知加工
每次产出一条原子笔记                 产出主题笔记、认知演变、成熟判断

用户发碎片 → 追问 → 笔记            碎片 → 诊断 → 连接 → 锻造 → 清晰认知
```

**Distill Engine 三层架构**：

```
用户的碎片输入（原料）
       ↓
  ┌────────────────────────────────┐
  │        Distill Engine          │
  │                                │
  │  Layer 1  诊断：这个想法有多熟？  │  ← 知识成熟度（方向 5）
  │              ↓                 │
  │  Layer 2  连接：碎片之间藏着什么？ │  ← 碎片聚合（方向 3）
  │              ↓                 │
  │  Layer 3  锻造：经得起推敲吗？    │  ← 认知挑战（方向 4）
  │                                │
  └────────────┬───────────────────┘
               ↓
  用户的清晰认知（产出）
               ↓
  成熟的认知 → 自动触发 V3 Express（输出）
```

---

#### Layer 1：诊断——知识成熟度评分

**解决的问题**：用户不知道自己哪些想法已经想透了、哪些还只是一闪而过的碎片。

**判断标准不是 AI 觉得熟不熟，是用户的行为数据说了算**：

| 成熟度 | 判断依据 | Crucible 的动作 |
|--------|---------|----------------|
| 种子 🌱 | 只聊过 1 次，没有应用场景 | 「这个想法你只提过一次，要不要再想想？」 |
| 生长中 🌿 | 聊过 2-3 次，或有 1 个应用场景 | 「这个想法你从两个角度聊过了，快成型了」 |
| 成熟 🌳 | 多次追问、多个场景验证、能用自己的话讲清楚 | 「这个认知已经成熟，可以输出了」→ 触发 V3 Express |

**数据来源**：crucible.db 中的对话记录 + 标签重合分析 + 笔记关联次数

**与 V3 Express 的衔接**：知识成熟度直接回答了「什么时候该输出」的问题——不是用户想写的时候，是知识成熟的时候。Crucible 主动告诉你「这个话题你已经想透了，该写出来了」。

---

#### Layer 2：连接——碎片聚合发现隐藏主题

**解决的问题**：用户聊了 10 次不同的想法，但没意识到其中 4 次都在讲同一个底层观点。

**场景举例**：
- 用户分别聊了「收藏不回看」「完美主义不敢写」「学了不用」「看了就忘」
- 这 4 条笔记的底层其实都在讲「知行不合一」
- 但用户自己没发现这个共同主题

**触发时机**：
- 某个标签下积累了 3+ 条笔记时
- 不同标签的笔记里出现了相似的观点表述时
- 用户主动问「我最近在想什么」时

**用户体验**：
> 「你最近聊了 4 个不同话题，但我发现它们都在讲同一件事——你似乎一直在想'怎么把知道的变成做到的'。要不要把这个提炼成一个完整观点？」

如果用户说「要」→ 启动**聚合式追问**：不是从零开始，而是把已有的笔记摆出来，让用户在它们之间找联系。产出一条**主题笔记**，比原子笔记高一个层级。

**笔记层级体系**：
```
主题笔记（聚合产出，Layer 2 的成果）
  ├── 原子笔记 1（单次对话产出）
  ├── 原子笔记 2
  └── 原子笔记 3
```

---

#### Layer 3：锻造——认知挑战

**解决的问题**：用户的观点可能只是「觉得对」，但从没被挑战过、没经过反面验证。

**这是 Crucible 最有差异化的能力**——市面上所有 AI 都在顺着用户说，没有 AI 在挑战用户思考。

**两种挑战方式**：

| 方式 | 什么时候用 | 例子 |
|------|----------|------|
| 反例挑战 | 用户给了一个全称判断 | 用户：「PM 一定要懂技术」→ AI：「你能想到一个不懂技术但做得好的 PM 吗？」|
| 逆向追问 | 用户的结论缺乏依据 | 用户：「AI 会替代 PM」→ AI：「这个判断是基于你看到的什么现象？如果反过来呢？」|

**设计原则**：
- **挑战不是抬杠**。目标是让用户自己发现漏洞，不是 AI 证明用户错了
- 挑战完之后，用户的观点可能不变，但变得更扎实了——这就是「锻造」
- 只在知识成熟度达到「生长中」以上时才触发挑战，种子阶段以鼓励为主
- 语气依然像朋友讨论，不是导师说教

**锻造后的产出**：笔记中新增「经过验证」标记，记录从哪些角度被挑战过、用户的最终判断是什么。

---

#### Distill Engine 的整体价值

| 维度 | 市面上的工具 | Crucible Distill Engine |
|------|------------|----------------------|
| 单次对话 | AI 帮你总结 | AI 追问你，让你自己想清楚 |
| 跨时间 | 无 | 追踪认知变化，发现隐藏主题 |
| 深度 | 顺着用户说 | 挑战用户思维，锻造认知 |
| 输出时机 | 用户自己决定 | 知识成熟了主动提醒输出 |

**一句话总结**：Distill Engine 不是帮用户「存知识」，是帮用户「炼认知」——诊断成熟度、连接碎片、锻造深度，最终让模糊的感受变成经得起推敲的清晰认知。

### To B 扩展可能性

Crucible 的核心能力（碎片输入 → AI 追问 → 结构化洞察）在企业场景中有天然适用性：

| 个人版（当前） | 企业版（扩展方向） |
|---|---|
| 个人碎片想法沉淀 | 团队会后复盘、调研洞察沉淀 |
| 一个人用 | PM 团队 / 研究团队协作 |
| Obsidian 本地库 | 企业知识库（权限隔离） |
| 飞书 Bot 入口 | 已在飞书生态，天然 To B |
| 个人知识成熟度 | 团队知识图谱 + 能力盲区分析 |

> 注：To B 扩展是架构预留方向，不是当前开发目标。当前核心逻辑解耦完成后，切换到企业场景只需要新增适配层 + 多用户权限，核心 Agent 逻辑（包括 Distill Engine）完全复用。

### 其他长期功能

- 知识图谱可视化（图形化查看笔记关联网络）
- 外部平台打通（微信读书划线、Readwise highlights 自动同步）
- 多用户支持（各自隔离的知识库，隐私优先）

### 2.3 技术架构迭代路径

Crucible 的开发过程本身体现了 AI Agent 技术的渐进式演化：

```
V1（基础 LLM API）        → V2（多 Agent 编排）
单次 Claude API 调用        Router → Socratic → Writer 三 Agent 串联
做追问                      消息路由 + 意图分类 + 流程编排

     → V3（Agent + 工具）         → V4（Agent + Harness）
       Agent 调用飞书 Bot、          OpenClaw 作为 Harness 管理
       Obsidian 写文件、SQLite       Agent 生命周期、会话上下文、
       入库、TAG_INDEX 上传          安全护栏；crucible.db 提供
                                    可观测性（自建评估层）
```

每一步都是因为真实产品需求驱动的迭代，不是为了堆技术栈。

### 2.4 数据驱动迭代策略（v0.3 新增）

V1-V2 功能已开发完成，下一阶段的核心不是加功能，而是**深度使用 + 数据积累 + 基于数据迭代**。

**行动计划**：
1. 每天用 Crucible 记录 2-3 个想法，持续两周，积累 30-40 条笔记
2. 两周后分析 crucible.db 数据：完成率、平均轮数、应用场景产出率
3. 根据数据调优 SOUL.md 中的追问策略
4. 对比调优前后的指标变化，形成「发现问题 → 改策略 → 验证效果」的闭环

**为什么这一步比加功能更重要**：一个被深度使用过、基于数据迭代过的产品，比一个功能丰富但从未被真正使用的产品更有说服力。

## 三、核心用户旅程

### 3.1 整体流程（CODE 四阶段）

CODE 方法论（Capture-Organize-Distill-Express）在产品内部作为架构框架运作，**用户感知不到阶段划分，只感知"在和 AI 对话，然后知识自动整理好了"**。

```
[用户] 发送内容给飞书 Bot
         ↓
[C - Capture]  Gateway 接收，识别内容类型
         ↓
[D - Distill]  Socratic Agent 启动追问对话（V1 核心）
         ↓ 对话结束
[O - Organize] 提炼核心洞察 → 生成原子笔记 → 写入 Obsidian → 建立双链（双链为 V2）
         ↓（V3）
[E - Express]  积累足够后，AI 聚合相关笔记生成草稿
```

### 3.2 工作应用场景（v0.3 新增）

Crucible 的追问 + 沉淀能力在日常工作中有三个高价值场景：

**场景 W1：会后复盘**
- 输入：「今天的需求评审效率太低了」
- Distill：追问哪个环节低？为什么？你怎么改？
- 输出：一条有行动方案的复盘笔记
- 价值：把「开完会就忘」变成「每次会议都有沉淀」

**场景 W2：竞品研究**
- 输入：看到一个竞品功能截图或文章链接
- Distill：追问它解决什么问题？你的产品缺这个吗？为什么？
- 输出：一条带判断的竞品洞察
- 价值：把「看了就过」变成「看了就有结论」

**场景 W3：用户反馈消化**
- 输入：客服转来一条用户吐槽
- Distill：追问这是个案还是共性？背后是什么需求？优先级怎么排？
- 输出：一条可以进 backlog 的需求分析
- 价值：把「反馈堆在那」变成「反馈变成决策」

### 3.3 个人使用场景（原有）

**场景 A：文章深度消化**

1. 用户在 X 看到一篇关于 AI Agent 架构的文章
2. 复制链接发送给飞书 Bot
3. Bot 回复："你发来了这篇文章，你最想搞清楚的是哪一点？"
4. 用户："我不太懂 Hub-and-Spoke 架构和普通微服务有什么本质区别"
5. Socratic Agent 追问 2-3 轮，引导用户用自己的话解释，并问："如果你要在当前项目里用这个架构，你会在哪个环节引入？"
6. 用户给出具体场景后，Agent 提炼一句洞察，生成笔记写入 Obsidian

**场景 B：随手想法捕捉**

1. 用户突然想到"产品的护城河不在技术，在用户行为习惯"
2. 直接发给 Bot
3. Agent 问："你是因为什么触发了这个想法？"→ "你觉得 Crucible 的护城河是什么？"
4. 用户在对话中逐渐想清楚，笔记自动生成

### 3.4 用户故事

**模块一：内容输入**
- 作为用户，我希望能把任何内容（链接/截图/一句话）发给 Bot，这样我不需要切换到其他 App 才能开始处理
- 作为用户，我希望 Bot 在 3 秒内开始响应，这样交互感觉自然不打断思路

**模块二：AI 追问（核心模块）**
- 作为用户，我希望 AI 每次只问我一个问题，这样我不会被多个问题淹没
- 作为用户，我希望 AI 的问题能锚定到我自己的场景（"你会在哪里用到这个"），这样思考感觉有意义，而不是在做作业
- 作为用户，我希望能随时说"先到这里"结束对话，这样我不会被强迫完成一次追问
- 作为用户，我希望每次对话结束后能看到一句洞察摘要，这样我知道这次思考收获了什么

**模块三：知识沉淀**
- 作为用户，我希望每次对话自动生成一条 Obsidian 笔记，这样我不需要手动整理
- 作为用户，我希望相关笔记之间自动建立双链，这样我能看到跨时间的知识关联（V2）
- 作为用户，我希望能通过 Bot 用自然语言检索历史笔记，这样随时能找回之前的洞察（V2）

### 5.2 Agent 分工

**Router Agent**
- 职责：接收所有飞书 Bot 输入，判断意图类型，路由到对应 Agent
- 意图分类：`内容消化`（文章/截图/想法）/ `知识检索`（V2）/ `草稿生成`（V3）
- 实现：Claude Haiku（速度快、成本低，适合分类任务）
- 输出：意图标签 + 结构化路由指令

**Socratic Agent（核心，自研）**
- 职责：通过 2-4 轮对话，引导用户从"模糊感受"到"具体个人应用场景"
- 追问三段式：
  1. **锚定**：找到用户最感兴趣的点（"你最想搞清楚的是哪一点？"）
  2. **深挖**：用具体场景替换抽象概念（"如果是你，你会在哪里用到这个？"）
  3. **收尾**：提炼洞察（"用一句话，你现在怎么理解这件事？"）
- 终止条件：用户给出个人应用场景，或用户主动说"先到这里"/"不想继续"
- 最大轮数：4 轮（超出后主动提示收尾，防止对话疲劳）
- 实现：Claude Sonnet（对话质量高，适合 Socratic 追问）

**Writer Agent**
- 职责：将对话产出格式化为标准 Markdown 原子笔记，写入 Obsidian
- 笔记模板：
  ```markdown
  # [洞察标题]
  
  **来源**：[原始内容链接/描述]
  **日期**：YYYY-MM-DD
  **核心洞察**：[一句话]
  
  ## 思考过程
  [对话中关键问答的摘要，保留用户原话]
  
  ## 个人应用
  [用户在对话中提到的具体应用场景]
  
  ## 关联笔记
  （V2 自动填入）[[相关笔记1]] [[相关笔记2]]
  ```
- 写入方式：直接写入本地文件系统（Obsidian Vault 目录），异步执行，不阻塞对话

**Knowledge Agent（V2 新增）**
- 职责：对每条新笔记做向量嵌入；检索相关历史笔记；生成双链
- 实现：LanceDB 本地向量数据库 + 语义相似度检索，取 Top 3 相关笔记

### 5.3 数据流

**V1 完整数据流**

```
用户发送内容（飞书 Bot）
  → OpenClaw Gateway 接收（WebSocket）
  → Router Agent 分类（~1s，Claude Haiku）
  → Socratic Agent 追问（2-4 轮，每轮 ~2s，Claude Sonnet，流式输出）
  → [对话结束] Writer Agent 生成 Markdown（~2s，异步）
  → 写入 Obsidian 本地文件（<0.5s）
  → 对话指标写入 crucible.db（SQLite，异步）
  → 飞书 Bot 返回洞察摘要确认
```

**V2 新增步骤（在 Writer Agent 之后插入）**

```
  → Knowledge Agent 对新笔记做向量嵌入（LanceDB）
  → 语义检索 Top 3 相关历史笔记
  → 在新笔记末尾补写 [[双链]]
  → 更新 Obsidian 文件
```

### 5.4 OpenClaw 作为 Harness 的角色分析（v0.3 更新）

> **Harness 定义**：Agent 的运行时管理层，负责生命周期管理、上下文管理、安全护栏、评估/可观测四个核心能力。

**OpenClaw 在 Crucible 中充当 Harness 的角色**，但并非完整覆盖所有 Harness 能力：

| Harness 能力 | OpenClaw 覆盖情况 | Crucible 自建部分 |
|---|---|---|
| 生命周期管理 | ✅ Agent 启动/结束、会话管理 | — |
| 上下文管理 | ✅ 单次对话内的连续性 | SOUL.md 定义跨会话的行为规则 |
| 安全护栏 | 部分（基础权限控制） | 追问最大轮数限制（SOUL.md） |
| 评估/可观测 | ❌ 无内置评估 | crucible.db + /stats（自建） |

**关键认知**：Crucible 的核心护城河不在 OpenClaw（它只是运行时），而在 SOUL.md 里的追问策略、路由规则、标签体系——这些是产品智慧，与平台无关。

### 5.5 核心逻辑解耦架构（v0.3 新增）

目前 SOUL.md 等核心文件仅存在于 OpenClaw 服务器上，与平台紧耦合。解耦后的目标架构：

```
项目 repo（版本控制）
├── agent_core/              ← 核心层：与平台无关的产品逻辑
│   ├── SOUL.md              ← 追问流程 + 路由规则 + 收尾格式 + 标签规则
│   ├── IDENTITY.md          ← 身份定义
│   ├── USER.md              ← 用户画像
│   └── HEARTBEAT.md         ← 定时任务规则
├── adapters/                ← 适配层：特定平台的接入方式
│   └── openclaw/            ← OpenClaw 部署脚本（当前使用）
├── sync-notes.py            ← 本地同步 + 评估入库
└── test_sync_notes.py       ← 测试
```

**解耦的核心价值**：
- 核心逻辑有版本控制，每次改追问策略都能 git diff
- 切换平台（Web 版、企业微信版）只需新增 adapter，核心逻辑复用
- 展示「Agent 逻辑与 Harness 分离」的架构思维

OpenClaw 在 Crucible 中负责：
- 通过飞书 Channel 适配器（`larksuite/openclaw-lark`）接收消息，WebSocket 长连接，无需公网 URL
- Gateway 管理 Agent 会话上下文（单次对话内的连续性）
- 将 Crucible 自研的三个 Agent（Router / Socratic / Writer）注册为 Skills 运行

Crucible 自研模块（OpenClaw Skills 形式接入）：
- Socratic Agent（核心追问逻辑，定义在 SOUL.md）
- Writer Agent + Obsidian 文件写入
- 标签系统 + TAG_INDEX（V2，替代原计划的向量检索）
- 评估数据采集（写入 crucible.db）

## 六、Distill 质量评估框架

> AI 产品与传统产品最大的不同：你永远在调模型行为，需要一套系统来判断 AI 是否在起作用。**不提前定义"好"，上线后只能靠感觉迭代。**

### 6.1 什么叫"追问成功"

**定义**：一次 Socratic 对话成功，当且仅当用户在对话中产出了**至少一个具体的个人应用场景**。

- 抽象表述（"这个概念很重要"）= **未成功**
- 具体应用（"我可以在 X 场景里用这个做 Y"）= **成功**

### 6.2 评估数据存储

所有评估指标写入本地 SQLite 数据库（`crucible.db`），与 Obsidian 同目录存放。

**核心数据表设计**：

```sql
-- 每次对话的汇总指标
CREATE TABLE conversation_metrics (
  id            TEXT PRIMARY KEY,   -- 对话 ID
  created_at    DATETIME,
  input_type    TEXT,               -- article / thought / image
  turns         INTEGER,            -- 实际对话轮数
  completed     BOOLEAN,            -- 是否走到收尾（vs 中途放弃）
  has_application BOOLEAN,          -- 是否产出了个人应用场景
  user_msg_chars INTEGER,           -- 用户消息总字数（参与深度）
  note_id       TEXT                -- 关联生成的笔记 ID
);

-- 每条笔记的生命周期指标
CREATE TABLE note_metrics (
  note_id       TEXT PRIMARY KEY,
  created_at    DATETIME,
  linked_count  INTEGER DEFAULT 0,  -- 被其他笔记链接的次数（V2 填入）
  searched_hit  INTEGER DEFAULT 0   -- 被检索命中的次数（V2 填入）
);
```

**观测方式**：通过飞书 Bot 发送 `/stats` 指令，触发统计查询并返回当周/当月核心指标摘要。

### 6.3 可观测信号（Proxy Metrics）

| 信号 | 含义 | 数据来源 |
|------|------|---------|
| 对话完成率 | 走到收尾 vs 中途放弃的比例 | `conversation_metrics.completed` |
| 平均对话轮数 | 轮数过少（≤1）追问太难；过多（>4）用户疲劳 | `conversation_metrics.turns` |
| 用户消息字数趋势 | 越说越多 = 追问有效；越说越少 = 追问太难 | `conversation_metrics.user_msg_chars` |
| 个人应用场景产出率 | 核心成功率指标 | `conversation_metrics.has_application` |
| 笔记被关联次数 | 知识真正融入体系的程度（V2） | `note_metrics.linked_count` |
| 7日/30日回访率 | 产品是否形成使用习惯 | 按 `created_at` 统计活跃天数 |

### 6.4 各版本评估重点

**V1 评估重点：追问机制是否有效**
- 核心问题：Socratic Agent 能否引导用户产出个人应用场景？
- 关键指标：对话完成率（目标 ≥60%）、个人应用场景产出率（目标 ≥50%）
- 迭代信号：完成率 <40% 或产出率 <30% 时，触发 Prompt 复查

**V2 评估重点：知识网络是否在生长**
- 核心问题：双链关联是否有意义？知识库是否在形成有效网络？
- 新增指标：平均每条笔记被关联次数（目标 V2 稳定后 ≥1.5）、检索命中率
- 迭代信号：笔记孤岛率（无任何双链的笔记占比）> 60% 时，检查向量检索质量

**V3 评估重点：输出是否真正发生**
- 核心问题：Express 草稿生成后，用户有多少比例真正发出了内容？
- 新增指标：草稿生成次数、草稿编辑率（生成后用户是否修改过）、最终发布率

### 6.5 核心指标目标（V1 阶段）

| 指标 | 目标值 | 触发复查的阈值 |
|------|--------|--------------|
| 对话完成率 | ≥ 60% | < 40% |
| 平均对话轮数 | 2-3 轮 | < 1.5 或 > 4 |
| 个人应用场景产出率 | ≥ 50% | < 30% |
| 笔记生成成功率 | ≥ 95% | < 90% |
| 7日回访率 | ≥ 40% | < 20% |

## 七、AI 失败模式 & Fallback 机制

> AI 产品必须设计 failure case，不能只有 happy path。

### 7.1 主要失败场景

| 失败场景 | 触发条件 | Fallback 策略 |
|---------|---------|--------------|
| 追问离题 | Socratic Agent 问了与输入无关的问题 | Router Agent 设置内容锚点并传入上下文；Socratic Agent 每轮前校验问题相关性 |
| 用户无法回答 | 用户连续两条消息表达困惑（"不知道"/"随便"） | 降级为提供选项："A 还是 B？"，或给一个例子让用户选择，降低回答门槛 |
| 内容无法解析 | URL 无法访问 / 截图模糊无法识别 | 提示用户补充文字描述，不静默失败；原始内容存入待处理队列 |
| 笔记写入失败 | Obsidian 文件锁 / 磁盘空间不足 / 同步冲突 | 重试 3 次；失败后将内容缓存为飞书消息草稿，告知用户手动确认后重试 |
| API 超时/限流 | Claude API 响应超时 > 10s 或返回 429 | 接收内容存入本地队列，API 恢复后自动重新处理，告知用户"正在处理，稍后回复" |
| 数据库写入失败 | crucible.db 损坏或写入异常 | 评估数据写入失败不影响主流程；记录到日志文件，不向用户暴露 |

### 7.2 用户体验原则

- **失败要告知，不能静默**：任何影响用户的失败都要发一条友好提示
- **内容绝对不能丢失**：无论哪个环节失败，原始输入必须保留
- **降级要优雅**：宁可给一个功能简单但稳定的体验，不给一个偶发崩溃的复杂体验

## 八、Prompt 迭代飞轮

### 8.1 问题发现信号

- 对话完成率骤降（Socratic Agent 追问质量下降）
- 用户频繁说"不知道"/"随便"（问题太难或太抽象）
- `has_application` 产出率低（没有引导到个人应用场景）
- 笔记内容雷同，缺乏个人化（AI 在输出模板而非真实洞察）

**定期抽样**：每周随机抽取 5-10 条对话，人工阅读判断追问质量，记录典型好/坏案例。

### 8.2 迭代策略

```
观察指标异常（完成率下降 / 产出率低）
    ↓
人工阅读 crucible.db 中的异常对话，定位问题环节
（是首问太难？深挖逻辑断了？收尾不自然？）
    ↓
修改对应环节的 Prompt
    ↓
新 Prompt 跑一周，对比修改前后的完成率和产出率
    ↓
选择更好的版本，在 prompts/ 目录下存档并注释修改原因
```

### 8.3 Prompt 版本管理

- 所有 Prompt 版本存入项目 `prompts/` 目录，每个版本标注：修改日期、修改原因、验证结果
- 原则：**不凭感觉改，每次修改必须有数据信号或抽样洞察支撑**

## 九、数据飞轮

### 9.1 飞轮逻辑

Crucible 天然具备**随使用积累而增强**的特性：

```
用户输入越多
    ↓
Obsidian 库越大（笔记数量↑）
    ↓
LanceDB 向量库越丰富（语义空间覆盖↑）（V2）
    ↓
RAG 检索的相关笔记越准确（V2）
    ↓
双链关联质量越高，知识图谱越有价值（V2）
    ↓
用户回访意愿越强（"每次找到惊喜关联"）
    ↓
用户输入越多 ←（闭环）
```

V1 阶段飞轮感知较弱（笔记是孤岛）；V2 引入向量检索后飞轮开始转动。

### 9.2 飞轮的临界点

| 笔记数量 | 飞轮状态 |
|---------|---------|
| < 30 条 | 关联稀疏，飞轮感知弱，用户依赖"追问体验"留存 |
| 30-100 条 | 开始出现有意义的跨时间关联，用户首次感受到"知识在连接" |
| > 100 条 | 知识图谱初步成型，搜索和关联价值显著，飞轮自驱 |

**产品启动策略**：V2 上线时，引导用户在前两周密集输入，帮助尽快越过 30 条临界点。

### 9.3 飞轮的边界

飞轮对**单个用户**有效（个人知识库越大越好）。由于 Obsidian 本地存储、数据不上云，不存在跨用户的网络效应飞轮——这是隐私设计带来的天然约束。

## 十、成本 & 延迟意识

### 10.1 API 调用成本估算（V1 MVP，个人使用）

| 操作 | 模型 | 预估 Token | 单次成本（估） |
|------|------|-----------|--------------|
| Router 分类 | Claude Haiku | ~500 tokens | ~$0.0003 |
| Socratic 追问（每轮） | Claude Sonnet | ~2,000 tokens | ~$0.006 |
| Writer 生成笔记 | Claude Sonnet | ~1,500 tokens | ~$0.0045 |
| **单次完整流程（3 轮对话）** | — | ~10,000 tokens | **~$0.03** |

**月成本估算**：每天 5 次输入 × 30 天 × $0.03 ≈ **$4.5/月**，个人使用完全可接受。

V2 新增向量嵌入成本：每条笔记嵌入 ~$0.0001，月增成本 < $0.5，可忽略。

### 10.2 延迟控制策略

**用户感知延迟目标**：首次响应 < 3 秒

| 环节 | 预估耗时 | 优化策略 |
|------|---------|---------|
| 飞书消息接收 | < 0.5s | WebSocket 长连接，无需轮询 |
| Router 分类 | ~1s | 使用 Haiku（速度快，分类任务足够） |
| Socratic 首轮回复 | ~2s | 流式输出（streaming），边生成边发送到飞书 |
| 笔记生成 + 写入 | ~3s | **异步执行**：对话结束后立即返回洞察确认，笔记写入在后台完成 |
| 评估数据入库 | < 0.1s | SQLite 本地写入，极低延迟 |

**关键设计原则**：笔记写入、双链生成、评估入库均为异步——用户收到洞察确认后，这些操作在后台静默完成，不让用户等待。

### 10.3 扩展性考量（V∞ 多用户场景）

当前 V1 设计为单用户本地架构，无扩展压力。若未来支持多用户：

- LanceDB 需改为支持多租户隔离的向量数据库（Pinecone / Weaviate）
- 每用户独立的知识库，数据严格隔离
- 引入请求队列（如 BullMQ）防止 Claude API 并发限流
- crucible.db 迁移为 PostgreSQL 以支持多用户查询

## 十一、技术选型理由

| 层级 | 选择 | 选择理由 | 被放弃的替代方案 |
|------|------|---------|---------------|
| 消息渠道 | 飞书 Bot | 用户已在用；飞书 CLI 200+ 命令覆盖全部需求；WebSocket 接收无需公网 URL | Telegram（国内访问不稳定）、微信（开放 API 限制极多） |
| Agent 框架 | OpenClaw | 开源 MIT；Gateway 控制平面成熟；飞书官方 Channel 适配器；Skills 插件系统便于自研模块接入 | 自建框架（重复造轮子，护城河不在消息路由层） |
| 沉淀端 | Obsidian 本地库 | Markdown 纯文本，数据完全归用户所有；Zettelkasten 双链天然适合知识关联；用户已有存量知识库 | Notion（数据在云端，API 限制多）、自建数据库（丢失用户已有 Obsidian 库） |
| 向量库（V2） | LanceDB（本地） | 本地部署，无需联网，隐私友好；支持 Rust/Python/Node；嵌入 + 检索一体，MVP 无需额外服务 | Pinecone（云端，隐私风险；MVP 过度设计）、Chroma（本地但生态较小） |
| AI 模型 | Claude Sonnet（主）+ Haiku（路由） | Sonnet 对话质量高，适合 Socratic 追问的多轮推理；Haiku 延迟低成本低，适合分类路由 | 单一模型（成本与质量无法兼顾）、GPT-4（国内访问不稳定） |
| 评估存储 | SQLite（crucible.db） | 本地零依赖；个人使用量级完全足够；与 Obsidian 同目录存放，便于管理 | PostgreSQL（V1 过度设计）、日志文件（无法结构化查询） |

## 十二、关键指标体系

### 12.1 北极星指标

**每月新增有效笔记数**

有效笔记 = `has_application = true` 的笔记（包含个人应用场景，而非空洞摘要）

这个指标代表用户真正在构建知识，而不只是在使用工具。

### 12.2 分层指标

**输入层（Capture）**
- 日均输入次数
- 输入类型分布（文章 / 随手想法 / 截图）

**处理层（Distill）**
- 对话完成率（目标 ≥ 60%）
- 平均对话轮数（目标 2-3 轮）
- 个人应用场景产出率（目标 ≥ 50%）

**沉淀层（Organize）**
- 笔记生成成功率（目标 ≥ 95%）
- 知识库总笔记数（飞轮临界点监控，V2 目标突破 30 条）
- 平均每条笔记被关联次数（V2 指标，目标 ≥ 1.5）

**留存层**
- 7日留存率（目标 ≥ 40%）
- 30日留存率
- 知识库周增长曲线（是否在持续生长）

### 12.3 反指标（需要立即复查的信号）

| 反指标 | 阈值 | 可能原因 |
|--------|------|---------|
| 对话完成率 | < 40% | Socratic Agent 追问质量问题，需 Prompt 复查 |
| 平均轮数 | < 1.5 | 用户在敷衍，没有真正参与思考 |
| 有效笔记率 | < 30% | AI 在生成模板而非个性化内容 |
| 7日留存率 | < 20% | 产品没有形成使用习惯，需重审核心体验 |

## 十三、附录：V1 MVP 验收标准

以下场景全部跑通，V1 视为完成：

- [ ] 用户发送一个 URL 给飞书 Bot，Bot 在 3 秒内开始流式响应
- [ ] Socratic Agent 完成 2-3 轮有意义的追问（问题与输入内容相关）
- [ ] 对话结束后，Obsidian `/crucible/` 目录出现对应原子笔记（格式正确，包含洞察摘要）
- [ ] 对话指标正确写入 crucible.db（轮数、完成状态、是否有应用场景）
- [ ] 用户发送随手想法（非文章），Bot 同样能正常追问并生成笔记
- [ ] 某个环节失败时，用户收到友好提示，原始内容不丢失
- [ ] 通过飞书 Bot 发送 `/stats` 能返回当周对话完成率和笔记数统计
