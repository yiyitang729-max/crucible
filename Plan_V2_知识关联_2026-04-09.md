# Crucible V2：收藏加工器 + 知识关联

## 一句话目标

Crucible 从「对话追问工具」升级为「收藏内容的加工器」——你只管收藏，它帮你消化、筛选、关联，告诉你「这堆东西里最值得你花时间的是什么」。

---

## V2 要解决的用户问题

**现状：** 每个人手机里都有大量「收藏了但没看」的内容。不是不想看，而是：
1. 收藏太多，不知道从哪个开始
2. 没有人帮你筛选优先级
3. 内容之间的联系看不见

**V1 做到了什么：** 用户主动发一个想法 → Bot 追问 → 生成笔记。用户必须「主动开聊」，而且每次只处理一条。

**V2 要做到什么：**
- 用户转发收藏内容给 Bot → Bot 自动消化内容（读文章/解析视频描述）
- Bot 追踪收藏模式（最近在关注什么方向）
- Bot 判断优先级，挑出最值得深入的内容，主动发起对话
- 所有笔记通过标签自动关联，形成知识网络

---

## V2 用户体验完整流程

### 日常收藏（每天随手做）

Tracy 刷到感兴趣的内容 → 手机分享到飞书 → 转给 Crucible Bot

Bot 收到后不追问，只回一句：
> 「收到，已加入待消化。」

（低打扰。收藏应该是零成本的动作，不能每收藏一条就被追问。）

### Bot 消化加工（后台自动）

Bot 自动做三件事：
1. **读内容**：访问链接，提取标题、摘要、关键信息
2. **打标签**：根据内容生成 3-5 个语义标签
3. **关联分析**：和历史笔记/历史收藏的标签比对，发现模式

### Bot 主动发起（每周 1-2 次）

积累一定量收藏后，Bot 主动发消息（不是提醒，是加工后的洞察）：

> 「你这周收藏了 6 条内容，主要集中在两个方向：
>
> 📌 **AI 产品能力**（3 条）— 有一篇关于能力模型的，和你之前聊过的「产品经理是首席用户」很相关
> 📌 **远程协作**（2 条）— 你之前提过远程办公效率更高，这两条可以作为补充
>
> 想从哪个方向开始聊？」

用户选一个方向 → 正常进入 Socratic 追问 → 生成笔记 → 该收藏的状态变为「已消化」。

### 对话中的被动关联

用户自己发来一个新想法时，Bot 生成笔记不仅关联历史笔记，也关联未消化的收藏：

> 「你的洞察是：XXX。
> 关联笔记：你 3 天前聊过 YYY。
> 另外，你上周收藏了一篇相关的还没聊过——要顺便消化掉吗？」

## 实现拆解（3 个子计划）

### Plan 5：标签系统 + 关联基础

**做什么：** 建立标签作为知识关联的底层机制。标签存在 crucible.db，是系统级数据，不绑定任何笔记软件。

**具体内容：**

1. **SOUL.md 笔记格式加标签**
   - Bot 生成笔记时额外输出 3-5 个标签
   - 两类标签：主题标签（讲什么）+ 思维模式标签（什么类型的思考）
   - Bot 通过读取 TAG_INDEX.json 知道已有哪些标签，优先复用

2. **crucible.db 新增表**
   ```sql
   CREATE TABLE note_tags (
       note_id   TEXT,
       tag       TEXT,
       tag_type  TEXT,  -- 'topic' 或 'pattern'
       PRIMARY KEY (note_id, tag)
   );
   ```

3. **sync-notes.py 扩展**
   - 解析笔记中的标签 → 写入 note_tags 表
   - 生成 TAG_INDEX.json（所有标签 → 对应笔记的索引）→ 上传到服务器
   - 计算关联（标签重合 ≥ 2 的笔记）→ 写入笔记文件的「关联笔记」章节
   - 更新 note_metrics.linked_count

4. **笔记格式变化**

   Bot 输出：
   ```
   **标签：** 远程办公, 沟通效率, 工作方式, 个人经验
   ```

   Obsidian 渲染（sync-notes.py 写入时转化）：
   ```yaml
   tags: [crucible, 远程办公, 沟通效率, 工作方式, 个人经验]
   ```

**验收：**
- [ ] Bot 笔记包含 3-5 个标签
- [ ] crucible.db 中 note_tags 有数据
- [ ] TAG_INDEX.json 生成且上传到服务器
- [ ] 两条标签重合的笔记在 Obsidian 里互相关联（双链）

### Plan 6：收藏入库 + 内容消化

**做什么：** 用户转发链接给 Bot 时，Bot 不追问，静默消化内容并入库。这是「收藏加工器」的核心管道。

**1. SOUL.md 新增路由类型**

Router 新增一类判断：

```
→ 收藏（bookmark）
- 用户只发了一个链接，没有附带想法/感受
- 或用户说了「收藏一下」「先存着」「记一下」
- 动作：回复「收到，已加入待消化。」（一句话，不追问）
```

与 content_digest 的区别：
- content_digest：用户发了想法或链接 + 看法 → 立即追问
- bookmark：用户只丢了个链接/说了「存一下」→ 静默入库，不追问

**2. Bot 后台消化内容**

收到链接后，Bot 调用工具读取链接内容（标题、摘要），生成：
- 内容摘要（一句话）
- 3-5 个标签
- 与已有标签的关联分析

这些数据写入 crucible.db：

```sql
CREATE TABLE pending_digests (
    id          TEXT PRIMARY KEY,   -- 飞书 message_id
    created_at  DATETIME,
    content_url TEXT,               -- 原始链接
    content_summary TEXT,           -- Bot 生成的一句话摘要
    tags        TEXT,               -- JSON 数组
    status      TEXT DEFAULT 'pending',  -- pending / digested / dismissed
    digested_at DATETIME,           -- 消化时间
    note_id     TEXT                -- 消化后关联的笔记 ID
);
```

**3. sync-notes.py 扩展**

- 从飞书消息中识别 bookmark 类型（Bot 回复了「收到，已加入待消化」的那些对话）
- 提取 URL、Bot 生成的摘要和标签
- 写入 pending_digests 表
- 将待消化内容的标签也纳入 TAG_INDEX.json

**验收：**
- [ ] 发一个链接（不带看法）→ Bot 回复「收到，已加入待消化」，不追问
- [ ] 发链接 + 看法 → Bot 正常追问（还是 content_digest）
- [ ] pending_digests 表有数据
- [ ] TAG_INDEX.json 包含待消化内容的标签

### Plan 7：收藏加工 + 主动对话

**做什么：** Bot 定期分析收藏模式，挑出最值得聊的内容，主动发起有价值的对话。

**1. 收藏模式分析（sync-notes.py 新增）**

每次同步时，分析 pending_digests 中 status='pending' 的内容：
- 按标签聚类：哪些方向收藏最多？
- 标记优先级：哪条和历史笔记关联最强？哪条收藏最久没消化？
- 生成 DIGEST_REPORT.json → 上传服务器

```json
{
  "updated_at": "2026-04-15 10:00",
  "pending_count": 8,
  "clusters": [
    {
      "theme": "AI 产品能力",
      "items": [
        {"id": "xx", "summary": "AI PM 能力模型", "age_days": 7, "related_notes": ["首席用户"]},
        {"id": "yy", "summary": "产品经理 vs 项目经理", "age_days": 3}
      ],
      "recommendation": "和你之前聊过的「产品经理是首席用户」很相关，建议从这个方向切入"
    },
    {
      "theme": "远程协作",
      "items": [...]
    }
  ]
}
```

**2. Bot 主动发起对话（HEARTBEAT.md）**

通过 OpenClaw 的心跳机制，每周检查一次 DIGEST_REPORT.json：
- 有待消化内容 → 组织成上面描述的「周报式」消息发给用户
- 用户选方向 → 进入 Socratic 追问 → 该收藏状态改为 digested

HEARTBEAT.md 任务：
```markdown
## 每周收藏消化提醒
- 频率：每周一 10:00
- 条件：DIGEST_REPORT.json 中 pending_count > 0
- 动作：读取 DIGEST_REPORT，组织消息，通过 feishu_chat 发送
```

**3. 对话中的被动关联**

用户聊新话题时，Bot 查 TAG_INDEX.json，如果发现某条 pending 内容标签重合 ≥ 2：
- 在收尾笔记中自然提一句：「另外，你之前收藏了一篇关于 XX 的，和这次聊的有关联——下次想一起聊吗？」
- 不强制，不打断追问节奏

**4. 用户控制**

- 用户对某条待消化内容说「不聊了」/「跳过」→ status 改为 dismissed
- 用户发 `/pending` → 显示当前待消化列表
- 用户发 `/digest` → 触发一次主动消化建议（不等每周自动触发）

**验收：**
- [ ] 收藏 3+ 条内容后，sync-notes.py 生成 DIGEST_REPORT.json
- [ ] Bot 每周发一次消化建议，按主题聚类，有优先级推荐
- [ ] 用户选方向后进入追问，消化完标记 digested
- [ ] 聊天时提到相关待消化内容
- [ ] `/pending` 和 `/digest` 指令正常工作

## 平台扩展性

标签和收藏数据全部在 crucible.db 里，不绑定 Obsidian。之后接其他平台：

| 平台 | 需要做的 |
|------|---------|
| Notion | sync-notes.py 加 Notion API 渲染层（标签→属性，双链→mention） |
| 飞书文档 | 已有 lark-drive 工具，加渲染层即可 |
| 其他笔记软件 | 只要支持 Markdown 导入或 API，加一个渲染器 |

收藏入口也不限于飞书：
| 入口 | 实现方式 |
|------|---------|
| 飞书转发 | 已有（V2 直接支持） |
| 微信 | 微信→飞书的消息转发（手动，但可行） |
| iOS 快捷指令 | 「分享」→ 调飞书 API 发消息给 Bot |
| 浏览器插件 | 点击→调 API 发链接给 Bot（V3 考虑） |

## 内容消化能力边界（重要）

Bot 能消化的内容类型：

| 类型 | 能力 | 实现方式 |
|------|------|---------|
| 文章/博客 URL | ✅ 能读取全文 | Bot 访问 URL 提取内容 |
| 公众号文章 | ⚠️ 部分可读 | 链接有时效，过期后无法访问 |
| 短视频（抖音/小红书） | ⚠️ 只能读标题和描述 | 无法看视频内容本身，但标题+描述通常够用 |
| 图片 | ⚠️ 取决于模型能力 | MiniMax 的视觉能力待验证 |
| PDF | ❌ 暂不支持 | 后续可加 |

对于视频类内容，Bot 的策略是：先读标题和描述打标签分类，追问时引导用户自己说出看了什么、有什么想法——核心价值不在于 Bot 看完视频，而在于帮用户把模糊印象变成清晰认知。

## 风险

1. **标签一致性**：Bot 可能同义词泛滥。解决：TAG_INDEX 提供已有标签列表给 Bot
2. **HEARTBEAT 可行性**：主动发消息依赖 OpenClaw 心跳，需要先验证
3. **内容读取失败**：部分 URL 防爬或过期。解决：读取失败时只存 URL 和用户的描述，不阻塞流程
4. **消化频率**：太频繁打扰用户，太少内容堆积。初始每周一次，可通过 `/digest` 手动触发

---

## 成功标准

| 指标 | 目标 |
|------|------|
| 笔记关联次数（平均） | ≥ 1.5（30 条笔记以上） |
| 笔记孤岛率 | < 60% |
| 标签复用率 | ≥ 50% |
| 收藏消化率（pending → digested） | ≥ 40%（30 天内） |
| 主动消化建议后用户继续聊的比例 | ≥ 30% |
