## 二、从痛点出发的核心洞察

### 五个原始痛点（按严重程度重新排序）
1. **只输入不输出** ← 最痛，是产品核心要解决的
2. 碎片化知识难以串联
3. 笔记软件过于分散
4. 微信读书沉淀差
5. Obsidian 熟练度低

### 深挖"只输入不输出"得到的关键发现

**表层**：没有输出机制

**真实原因**：工作环境导致的完美主义 ——「没想完整不敢输出」

**行为模式**（用户已有的隐性 workflow）：
```
看到文章 → 收藏（但不看）→ 有时截图转发给 GPT/Gemini/OpenClaw → 得到解释 → 理解停在聊天记录里消失
```

**真正的断点**：不是"存进去"，而是**存进去之后什么都没发生**

### "真正理解"的标准
用户自己给出的案例：看到 CodeX 变成 Claude Code 插件 → 理解了它对 AI Builder 的意义 → 觉得"可以用起来了"

**结论**：真正理解 = 从抽象概念 → 找到在自己场景的具体应用

## 三、产品方向的关键转变

### 原始设计 vs 脑暴后的修正

| 维度 | 原始设计（需求前讨论记录） | 脑暴后修正 |
|------|--------------------------|-----------|
| 核心角色 | The Warden（守门人）——不思考不准存 | 思考伙伴——先接住，再追问 |
| 入门门槛 | 阻断式（高门槛） | 低门槛（任何碎片都可以进来） |
| 深度发生的时机 | 入库前 | 入库时（对话过程中） |
| 用户感知 | 主动感知 CODE 四阶段 | 只感知"在和 AI 对话"，CODE 在后台 |

### 为什么要改

用户有完美主义倾向，"阻断式守门人"会触发完美主义，反而让用户更不敢输入。需要先降低门槛，在对话中自然地逼用户深度思考。

## 四、产品定位（最终版）

**一句话**：一个 AI 思考伙伴，让你移动端的每个碎片想法不再消失——通过对话式追问变深，自动沉淀为 Obsidian 里的结构化知识，最终能产出输出。

**产品本质**：拦截用户「已有的把文章发给 AI 问的行为」，让这个行为的产出不再消失在聊天记录里，而是积累成可用的个人知识库。

## 五、架构决策

### 技术方案：B + C 的结合

- **B（对话驱动）**：入口永远是"和 AI 聊"，用户不感知流程，接近已有行为习惯
- **C（原子化 Zettelkasten）**：每条输入变成一个原子笔记，AI 自动建立双向链接

### 技术选型

| 层 | 选择 | 理由 |
|----|------|------|
| 入口/渠道 | 飞书 Bot | 用户已在用；飞书 CLI 支持完整操作；无需公网 URL |
| Agent 框架 | OpenClaw | 开源；Hub-and-Spoke 架构；内置混合记忆（向量+FTS5）；飞书官方集成 |
| 沉淀端 | Obsidian 本地库 | Markdown + 双链；本地隐私；用户已有库 |
| 核心 AI | 自研 Socratic Agent | 这是护城河，不外包 |
| 知识关联 | RAG + 向量检索 | 语义关联已有笔记，跨时间串联单点想法 |

### CODE 四阶段在产品里的映射

```
C (Capture)   → 飞书 Bot 接收任何输入（文章/截图/一句话）
O (Organize)  → 自动生成原子笔记 + 双链，写入 Obsidian（用户不感知）
D (Distill)   → Socratic Agent 追问对话（产品核心，用户唯一感知到的 AI）
E (Express)   → V2：AI 聚合相关笔记生成草稿（笔记→推→文章，渐进式）
```

用户感知不到 CODE，只感知"在和一个懂我的 AI 对话，然后知识自动整理好了"。

## 六、Socratic Agent 的设计原则

**不是守门人，是追问者**

1. 任何输入都被接住（低门槛）
2. AI 的追问方向：「这对你意味着什么？」「你会在哪里用到这个？」
3. 不需要用户"完全想清楚"，部分洞察也有价值
4. 对话结束后，AI 提炼一句核心洞察存档

**追问的终止条件**（待细化）：
- 用户给出了一个具体的个人应用场景
- 或用户主动说"先到这里"

## 七、亮点：AI 效果评估框架（Portfolio 加分项）

这是大多数 PM 没认真想过的部分，也是面试时的差异化。

**核心问题**：怎么知道 Socratic Agent 追问得好不好？

**待设计的评估维度**（设计文档里需要展开）：
- 追问成功的定义（用户产出了具体应用场景 vs 用户中途放弃）
- 可观测信号（对话轮数？用户输出字数变化？洞察被关联次数？）
- 如何迭代 prompt（基于哪些信号判断追问质量下降？）

**为什么必须从设计阶段就定义**：不提前定义"好"，上线后只能靠感觉迭代，无法证明 AI 在起作用。

## 八、MVP 范围

**V1（MVP）**：Capture → Distill → Organize
- 飞书 Bot 接收输入
- Socratic Agent 追问对话
- 自动生成原子笔记写入 Obsidian
- 基础双链关联

**V2**：
- Express（AI 草稿生成）
- 收藏焦虑解决（定期从收藏夹唤起未处理内容）
- 更完整的知识图谱可视化

## 九、市场背景（简版）

- AI 笔记/PKM 市场 2025 年约 $6 亿，CAGR ~19%，2035 年预计 $35 亿
- 市场高度分散，无玩家超过 18% 份额
- 所有主流工具（Mem、Notion AI、Readwise、flomo）都在解决 Capture + Organize
- **Distill 层是真空**——没有工具主动帮用户把模糊感受变成清晰认知
